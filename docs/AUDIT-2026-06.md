## 🔴 HIGH

### H1. WFP-фенс блокирует xray.exe → XHTTP-ноды мертвы при kill-switch
`windows/runner/kill_switch.cpp`. Фенс пермитит app-id только sing-box (+loopback+TUN), но **не `xray.exe`**. Любая нода через xray-бридж (XHTTP, Reality-over-XHTTP) при включённом фенсе теряет весь трафик — «защита» = тотальная темнота именно на твоих лучших транспортах.
**Fix:** добавить app-id `xray.exe` (всех бриджей) в permit-список фенса; брать пути из того же места, что и спавн бриджей. **Test:** unit на сборку permit-списка (содержит и sing-box, и каждый xray-бридж).

### H2. Недоверенный импорт автоконнектится без согласия
`import_actions.dart` → `applyImport` = `select` + `restart()`. Deeplink/QR/drag из чужого Telegram → трафик молча идёт через ноду атакующего (MITM). Для твоей аудитории honeypot = слежка.
**Fix:** развилка trusted (in-app диалог/файл-пикер) vs external (argv-deeplink/QR-картинка). Внешний путь: импорт без автоконнекта → **превью узла** (host/протокол/порт/SNI + insecure-badge) → Connect по явному подтверждению. **Test:** external import → `select` без `restart`; preview содержит host+флаги.

### H3. system-proxy режим без kill-switch (fail-OPEN + DNS-утечка)
`core_controller.dart`. Фенс только в TUN. В дефолтном system-proxy при смерти ядра/реконнекте не-proxy-aware трафик и DNS идут напрямую.
**Fix:** либо WFP-фенс и для system-proxy (block-all-except core app-id + loopback), либо честный UI-дисклеймер «system-proxy не leak-proof, для защиты — TUN+kill-switch». Лучше — фенс в обоих режимах. **Test:** при `status==error` в system-proxy фенс держит/прокси не восстановлен.

### H4. TUN kill-switch fails OPEN на give-up и при сбое установки
`core_controller.dart`. После исчерпания реконнектов и на терминальной ошибке фенс снимается (fail-open) без согласия; если фенс не установился — TUN-коннект идёт **без защиты**.
**Fix:** политика fail-closed-до-явного-Stop (с понятным UI и кнопкой «разблокировать», чтобы не было lockout); если включён kill-switch и фенс не встал — не подключаться молча, а предупредить/блокировать. **Test:** give-up оставляет фенс; fence-install-fail → коннект не идёт молча.

### H5. insecure:true нода автоконнектится без MITM-предупреждения
`proxy_node.dart` / `profiles_sheet.dart`. Detection есть (бейдж), но это пассивный значок; импорт+коннект проходят без активного предупреждения.
**Fix:** на импорте/коннекте insecure-ноды — модалка «сертификат не проверяется, нода может читать трафик» + явное согласие. **Test:** insecure-нода → коннект требует подтверждения.

## ТЕСТ-ПЛАН (что покрыть)

Главный технический долг — **safety-critical логика без тестов**. Сначала развязать швы:

1. **Вынести seam'ы:** `_tunnelHealthy()` и `_directNetworkUp()` создают свой `HttpClient`/`Socket` → за инъектируемый интерфейс (вместе с `ClashApi`). Тогда тестируемо без сети.
2. **Watchdog/каскад (мок ClashApi+probes):** сеть-down → НЕ каскадить; сброс эпизода по устойчивому recovery; IP-блок dead-end; выбор семейства по диверсити.
3. **Kill-switch lifecycle:** stop/restart/crash/give-up/wake → ассерты `_proxyActive`/состояния фенса; инварианты `_onExit` (fail-closed).
4. **Leak-shape конфигов:** `withTun` порядок split/force правил; `desyncOnly` reject QUIC/UDP-443; permit-список фенса (содержит xray-бриджи — H1).
5. **Deeplink:** `importablePayload` — двойной decode `%`, каждая схема, фильтр флагов.
6. **Config-migration в CI:** прогон реального `sing-box check` (есть `tool/verify_store.dart`) как тест, не только in-process корпус.
