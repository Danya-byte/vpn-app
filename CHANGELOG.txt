1.0.5 (26.06.26)

- A redesigned "liquid glass" look. The frosted panels now float above a deeper, living backdrop with real depth — a soft shadow under each panel instead of looking painted flat onto the background; the big Connect button glows in the app's green and blooms a wider halo when connected; and the drag handle at the top of the pop-up sheets is now clearly visible instead of a faint hairline. The window itself has rounded corners now, and colours, text sizes and corner radii were unified into one consistent scale across every screen.
- Control the VPN from the system tray. Right-click the tray icon (when the app is closed to the tray) to Connect / Disconnect without opening the window — the menu label follows the live state. And a tray action is no longer silent: the tray icon's tooltip shows the current state, and a notification tells you when it connected, disconnected, or — the important part — exactly WHY a connect failed (the core's own error text), so a tray connect that doesn't work no longer leaves you guessing with no feedback. Clicking the notification opens the window to see the details.
- The first-run protection-mode chooser can no longer be dismissed by a stray click onto the leak-prone default — it asks until you actually pick, so a newcomer never lands on the weaker mode by accident.
- Junk server links are rejected up front instead of importing "green" then silently failing to connect: a vmess / Shadowsocks / Clash entry with no server, no credentials, or an out-of-range port is now dropped at import with the reason, not added as a dead node.
- Hardening, from a whole-app audit (every screen and every function, reviewed and adversarially double-checked):
  - A corrupt or truncated settings file no longer silently resets EVERY toggle (kill-switch, bypass, saved credentials) to defaults — settings are now mirrored to a backup and recovered, and the live store can't be left empty by a crash mid-save.
  - Importing someone's shared settings bundle with empty lists no longer wipes your own split-tunnel / force-VPN / custom routing rules.
  - The live speed graph and the connections list no longer freeze or blank out on a single malformed frame from the core.
  - Smart mode with FakeIP no longer routes Russian domestic sites (gov / banks) out the foreign server when the geo rule-sets aren't ready — FakeIP only engages with the domestic carve-out in place.
  - Bridged AmneziaWG tunnels no longer go silent after a NAT timeout (a keepalive is now sent).
  - Encrypted ClientHello (ECH) retries within a session instead of giving up for good after one blocked DNS lookup.
  - "Create your own server": the two-hop relay chain no longer silently carries no traffic (a handshake-mode mismatch on the relay hop is fixed), an IPv6 server address now makes a valid importable link, and the flow no longer says "added" when nothing was actually added.
  - A hostile imported config can no longer crash the whole connect with a type error — it skips the one bad node instead.
- Accessibility: the switches, the mode picker, the Connect button and the bottom navigation now announce themselves and their state to screen readers, and the connections panel distinguishes "connecting", "error", "not connected" and "no connections" instead of collapsing them into one message.
- Telegram without a server (new toggle under "Unblock without a server") — restores Telegram CALLS and steadies messaging on-device, no foreign server. Russia blocked Telegram/WhatsApp calls in Aug 2025 by matching a packet signature (the STUN "magic cookie") and dropping it; the app's bundled packet engine now sends a decoy that fools that filter on the way out, scoped strictly to Telegram's own server addresses, so calls connect again and the MTProto handshake is desynced for steadier text. Needs administrator (it loads a network driver), like the existing bypass — it shares the same engine. HONEST LIMITS, shown in the app: media (photos/video) stays throttled because that's an IP-speed limit, not a signature — only a server (the built-in Telegram-to-tunnel routing) restores it; and the southern regions / full shutdowns block Telegram by IP, which no on-device trick can beat. The call fix also only covers your side of the line — if the person you're calling is behind the same block, their side can still degrade.
- Native ECH masquerade — hide which site you're really reaching, the way Chrome does. Turn on Encrypted ClientHello (Settings → Advanced → ECH) and the app now finds each server's published ECH key automatically over encrypted DNS and encrypts the real TLS server name inside the handshake; on the wire only a harmless "cover" name (for a Cloudflare-fronted server, a Cloudflare name) is visible. No separate program, no server setup, nothing to paste. It works best for servers behind Cloudflare (or your own ECH endpoint); for a plain server that publishes no ECH it changes nothing and never blocks or slows the connection (the lookup is best-effort and times out safely). Reality servers are left untouched — they already hide their own name a different way. (Honest scope: it only does anything when the server actually publishes an ECH key in DNS — for a plain VPS that publishes none it's a no-op, not a magic unblock.)
- Telegram without a server, natively (Settings → Advanced). A bundled local engine runs a Telegram proxy on your machine and bridges it to Telegram's own web gateway over a WebSocket — but reaching the gateway through the few Telegram addresses that Russia's filter leaves un-throttled, and disguising the TLS handshake as your real browser (it captures your browser's fingerprint once, automatically). So Telegram — messages AND media — rides an un-throttled path that looks like ordinary browser HTTPS, with no foreign server. You flip the toggle and tap "Open in Telegram" (the link adds the local proxy in one tap). The base bridge needs no administrator; an optional "Calls" switch adds a packet-level desync for voice/video (that one needs administrator). Honest scope: it works from inside Russia (the un-throttled gateway is only clean from under the national filter), it's about reliability + media rather than raw speed, and a hard regional IP-shutdown still needs a real server.
- Hardening, from a deep self-audit of this release: a malformed or hostile imported config can no longer crash the import-safety preview (it degrades to a warning instead of a red error screen); the generated domestic-relay chain's exit server no longer mandates a handshake mode the client doesn't send (which silently broke that hop); the "is Telegram already routed?" check now recognises every rule form, so it never overrides your own Telegram→direct rule; dead DNS-over-HTTPS fallbacks were replaced with working ones; and the pre-connect latency probe won't fire a stray direct dial once the tunnel is already up.
- The native serverless-Telegram engine now reuses your EXISTING local proxy instead of quietly creating a second one: it runs on the same port with the same secret your Telegram already trusts, and attaches to an instance you're already running rather than spawning a conflicting duplicate — so "Open in Telegram" always adds the proxy that actually works, and the one-time browser-fingerprint capture runs in the background instead of holding Telegram down while it waits.
- Settings de-cluttered, duplicate Telegram switches merged. There used to be THREE "Telegram without a server" controls (the experimental in-app bridge, a winws Telegram sub-toggle, and the native engine) — now there is ONE: the native engine. The experimental bridge was removed entirely, and the winws Telegram sub-toggle was removed too (its only real job, restoring calls, is already covered by the native engine's own "Calls" switch — same packet-level fix, one place). Also: the Anti-DPI switch now shows as locked-on when "Hard-network mode" is already forcing it, instead of sitting there as a second contradictory toggle; and the advanced multiplex options (carrier protocol, padding) appear only when Multiplex itself is on, rather than as dead controls.
- Settings honesty pass (from a per-function audit of every control). The native Telegram engine no longer claims "Running" when the background process actually died or was never bundled — it now confirms the proxy is really listening before reporting success, drops the status the instant the engine exits, and shows an explicit "unavailable" line instead of an endless "checking…" spinner. The Telegram "Calls" switch is disabled with a "needs administrator" prompt when you're not elevated, instead of flipping green while silently doing nothing. Switching to TUN mode while connected without admin now offers to relaunch elevated rather than tearing your working connection into an error. And the server-less bypass status no longer tells you to "connect" to start an engine that actually runs the moment you flip its toggle.
- Pre-release hardening (full-diff audit). Fixed a regression where turning on Encrypted ClientHello had quietly stopped working inside Russia — it now keeps the in-tunnel DNS fallback that still resolves there. A malformed or hostile imported config can no longer abort the WHOLE connect (every TLS field is type-guarded now, not just one), and a single bad WireGuard endpoint port no longer throws away every other server in the same import. A subscription auto-refresh no longer silently deletes a server you pinned a certificate to (which would have undone the man-in-the-middle protection) and now restarts the tunnel if it pruned the very server you were connected through. Connecting from the tray to a certificate-unverified server you've never confirmed is now refused with a notification instead of silently building a MITM-able tunnel. Server pings no longer flash "blocked" when a probe is cut short by connecting; the native Telegram engine cleans up a process that failed to bind; and a settings/profile save blocked by antivirus mid-write is recovered on the next launch instead of lost.
- Desktop apps that route through the system proxy (Chrome / Edge / Electron apps — e.g. the Claude desktop app) no longer fail with ConnectionRefused / ERR_PROXY_CONNECTION_FAILED under the VPN. This bit hardest when you ALSO run another local VPN like Hiddify: Hiddify points the Windows system proxy at its own local port (e.g. 127.0.0.1:12334), and the app used to leave that in place. In TUN mode — where the tunnel already captures every app transparently — that leftover proxy hijacks proxy-aware apps onto a broken path: through Hiddify and back into our own tunnel (a pointless double-hop), or onto a dead port if Hiddify is closed (ConnectionRefused). Now, on every TUN connect the app SUSPENDS any local (loopback) system proxy so those apps fall straight through the tunnel like everything else; your own proxy is backed up and put back on disconnect — only if it's still alive, so a dead one is left off instead of stranding apps on it. A real upstream proxy (a company proxy on a real host) is never touched. Clearing the proxy also broadcasts the Windows "Internet Settings" change those apps refresh on (the WinINET notification alone doesn't reach Chromium-based apps), so they pick up the change without a restart. Together this makes the one-time manual "turn the Windows system proxy off" step from the 1.0.4 notes automatic.
- "Connects but the internet doesn't work" — a cluster of connectivity fixes from a deep audit against the working Hiddify reference (same server). (1) DNS was the big one: foreign names resolved over a single connection through the tunnel that head-of-line-stalled and made lookups take 6–34 SECONDS while real traffic flowed in 160 ms — so everything FELT dead though the tunnel was fine. Foreign DNS is now sent as UDP over the tunnel, so a burst of lookups resolves in parallel instead of queuing on one stalling connection; the DNS cache is persisted to disk (like Hiddify) so reconnecting starts warm instead of re-resolving everything slowly; and Russian sites resolve DIRECTLY off-tunnel. The tunnel still hides DNS from DPI, and the RU carve-out is always on (no geo rule-set needed). (2) VLESS connections now carry UDP with xudp packet-encoding (like Hiddify), so QUIC / HTTP-3 sites and voice stop hanging while plain pages loaded. (3) In TUN mode IPv6 now fails over to IPv4 INSTANTLY instead of sitting in a multi-second timeout first (every exit is IPv4 here), so first page-loads are quick again — still no IPv6 leak. (4) The auto-cascade no longer abandons a solid Reality (TCP) server to hop onto a throttled Hysteria2 (UDP/QUIC) one for a mere latency blip: QUIC is now used only when a 16 KB-freeze actually needs it, and a normal degrade prefers another TCP/Reality server. Together these are why a tunnel could be "up" yet carry no usable traffic.
- Copy buttons now confirm on the control itself. Tapping a copy affordance — the config viewer and node dialogs, the SOCKS5 proxy address, the activity log, the server-diagnostics report — flashes a green check for a moment instead of looking like nothing happened, so you can see the copy landed without watching for a toast.
- AmneziaWG support. The app now bundles the userspace AmneziaWG bridge (awg.exe), so an imported AmneziaWG config — the obfuscated WireGuard with jc/jmin/s1–s4 — is carried through that bridge instead of being detected-and-failed. Plain WireGuard already worked natively; this adds the Amnezia-obfuscated variant. If the bridge binary isn't present it still fails closed, never pretending to connect.
- The native server-less Telegram toggle now frees its port the instant you turn it OFF. Before, the local proxy could be left holding its port (the process was asked to stop but not waited for, and a quick re-enable could re-attach to the still-dying instance); now OFF kills the whole process tree and waits for the port to actually release — and if the app had attached to a tgcore you started yourself, OFF stops that too (only ever a verified tgcore, never an unrelated program that happens to hold the port).
- More hardening, from an extra-high-effort review of the entire release diff (every changed file, each finding independently re-verified):
  - Connecting from the tray to a certificate-unverified server you've never confirmed is now refused even when no server is explicitly selected — the gate checks the exact server the connect would use, closing a dangling-selection gap — and it raises the window so you can give consent.
  - A single click on the tray icon restores the window again (a notification-format change had quietly broken icon clicks, leaving only right-click → Show).
  - In TUN mode the app also suspends an IPv6-loopback system proxy (e.g. [::1]:port) from another local VPN, not just an IPv4 one, so a proxy-aware app can't be hijacked over the tunnel through a leftover v6 proxy.
  - Network diagnostics no longer misreads a slow connect as an active refusal (so a sluggish link can't fabricate a scary "offline"), and still counts a blocked site as reached through the tunnel when it answers with a server (5xx) error.
  - The "is Telegram already routed?" check no longer false-matches an unrelated domain or a proxy group merely named "Telegram", so the Telegram unblock routing isn't silently skipped.
  - A Shadowsocks "none"/"plain" entry (no encryption, empty password) imports again instead of vanishing; a WireGuard config with an empty endpoint host is dropped at import instead of importing "green" then failing to connect; and an out-of-range WireGuard keepalive is clamped instead of breaking the bridge.
  - Consent you give for an insecure node mid-connect is no longer cancelled by a connect/disconnect status race, and the in-app segmented tabs now expose their tap action to screen readers.

1.0.4 (16.06.26)

- Fewer reconnects still: the "is the tunnel alive?" check now retries once before it declares the tunnel dark, so a single stalled probe on a flaky mobile operator no longer triggers a needless server hop (which used to cut your live connections). Builds on 1.0.3's "trust real traffic" fix.
- Telegram (and other long-lived connections) stop dropping constantly: the auto-server picker no longer cuts existing connections when it switches to a faster server — only new connections move, while your live Telegram session keeps running. Imported configs that forced this off are corrected on load, and the built-in auto-pool is steadier (it won't re-pick on a tiny latency wobble).
- Importing an AmneziaWG config now warns you upfront: AmneziaWG's obfuscation needs a separate bridge that isn't installed, so the config would fall back to plain WireGuard — which an AmneziaWG server rejects (it silently wouldn't connect, and the ping never measured). A normal WireGuard .conf still works as-is; the warning just tells you why an AmneziaWG one won't, instead of leaving you guessing.
- Fix (ERR_PROXY_CONNECTION_FAILED in the browser): on disconnect the app now restores exactly the proxy you had before connecting — including a local proxy you run yourself (cntlm, Burp, Fiddler, a local Shadowsocks…). It no longer mistakes your own 127.0.0.1 proxy for its own and drops it; only its OWN port (127.0.0.1:2080) is treated as the app's, so your real proxy is backed up and put back faithfully. And when the app disconnects with no saved proxy to restore and the system is still pointed at the app's own dead port, it switches that off so the browser falls back to a direct connection instead of erroring. (If a DIFFERENT tool left a dead local proxy behind — e.g. "http://127.0.0.1:12334" from another VPN that crashed — turn the Windows system proxy off once: Settings → Network & Internet → Proxy.)
- Telegram's web app (web.telegram.org) is added to the server-less DPI-bypass site-list, so on operators that only SNI-block it (its IP still reachable) it opens with no server at all. (Telegram's desktop/mobile apps stay IP-blocked and still ride the tunnel via the built-in native unblock.)
- A malformed or hostile config can no longer crash the import preview: a bad "endpoints" section is ignored instead of throwing, so the preview still renders the server and still refuses to auto-connect.
- Refreshed the bundled Russia site-list (geosite category data) to the latest upstream, pinned to an immutable commit so the release build's checksum can't silently drift.
- New: measure each server's latency BEFORE connecting — tap the gauge icon in your server list (while disconnected) and every server gets a colour-coded ping (green/amber/red), so you can pick a fast, reachable one without trial-and-error. It probes safely (resolves names over encrypted DNS, so a poisoned answer can't paint a dead server green) and marks UDP servers (Hysteria2/TUIC) that a quick TCP check can't measure.
- New: pin a self-signed server's certificate (Hysteria2/TUIC). A server with no real certificate normally forces you to accept "no verification" — an on-path interception (MITM) risk, flagged amber. Now you can paste the server's certificate once to PIN it: the connection is then verified against exactly that certificate, the amber warning turns into a green "verified" badge, and a wrong paste is reversible (tap the green shield to un-pin).
- Share a single server straight from its row (the new share icon), and "share for any app" now hands out ALL the servers inside a config, not just one.
- The app now LEARNS which transport survives on YOUR network: when it auto-switches to a working transport during blocking, it remembers — and next time tries that one first, so recovery on your operator gets faster over time (remembered across restarts). The switch stays seamless: your live Telegram/downloads aren't cut.
- More resilient DNS: when the usual encrypted-DNS server (1.1.1.1) is itself blocked in Russia, the app now races several others (Google / Quad9 / …) in parallel, so name resolution — and the new latency check — still work under heavy blocking.
- Server-less DPI bypass: a one-tap "Try next method" on the bypass card cycles through the desync presets (and rotates the decoy site name) when a site still won't open — no more guessing which preset your operator needs.
- Telegram, incl. calls, in any mode: Settings → connection mode now shows a tap-to-copy SOCKS5 address (127.0.0.1:2080). Point Telegram's own SOCKS5 proxy there (Telegram → Settings → Advanced → Connection type → SOCKS5) and its messages AND voice/video calls ride the tunnel — even in system-proxy mode, where the Windows proxy alone is TCP-only (so calls otherwise wouldn't go through). A transport hop never cuts a live Telegram call (the picker keeps existing connections pinned).

1.0.3 (06.06.26 - 12.06.26)

- Fix: far fewer needless reconnects. The "is the tunnel alive?" watchdog now trusts real traffic — if data is actually flowing through the tunnel, it won't declare it "dark" and hop to another server just because a single background health-check timed out (a flaky mobile operator can stall that one probe while your real browsing works fine). Each unnecessary hop used to cut your live connections, so this also stops the connection drops that came with them.
- Cleaner, smoother look: the on/off switches now glide with the same spring as the bottom navigation bar, the frosted "liquid glass" panels are more translucent across the whole app, and the big Connect button lost the little white highlight dot that sat on top of it — less visual noise, more polish.
- New "Advanced" section in Settings, collapsed by default so it never clutters the main screen — expert transport knobs for those who want them, invisible to everyone else: choose the TUN network stack (gVisor / system / mixed), pick the multiplexing protocol (h2mux / smux / yamux) with optional stream padding, enable Encrypted ClientHello (ECH) to hide the TLS server name on non-Reality servers, set a DNS Client Subnet (ECS) for better CDN locality, and opt into TCP Fast Open or Multipath TCP. Every knob defaults to the previous behaviour, so nothing changes unless you choose it — TCP Fast Open is clearly flagged as risky on Russian mobile networks. A malformed DNS Client Subnet (a typo'd IP / bad prefix) is rejected as you enter it, so it can never take a working tunnel down.
- Hysteria2 port-hopping: a server link that carries a port range now rotates across those ports automatically to slip past port-based throttling, instead of being pinned to a single port. An out-of-range or reversed range (e.g. 70000-80000, 8443-443) is ignored and the single port is kept, so a typo'd link can't leave a silently-dead node.
- New: share your setup as a link (Profiles → ⋮ → Share). "For any app" makes a standard link any VPN client can import; "With my settings" makes a link that — in this app — also carries your DPI-bypass + per-app routing setup, so a friend gets your whole working configuration in one paste. Importing a shared link always previews the server and asks before applying any settings or connecting, so a link from an untrusted chat can never silently change your protection or route you through a hostile server. The protection bundle never includes your private data (cloud-sync credentials, custom DNS, accepted-risk servers). Links shared today keep importing in future versions, and older app builds keep importing them.
- Fix: a typo'd custom routing rule (an absurdly long CIDR prefix) could silently reset EVERY setting to defaults on the next launch — the value is now rejected as you type and can never wipe your settings.
- Hardened share-link import end-to-end: a maliciously crafted compressed link can no longer freeze the app (decompression is strictly capped); the "with my settings" preview now shows the same red flags as a normal import (a no-certificate-check server, a config that routes everything outside the tunnel, a widely-blocked transport — plus the exit servers themselves); a shared auto-update URL is accepted only over https; and after you confirm, the app now tells you if the imported server failed to come up or carries no traffic, instead of a green "Added" over a dead tunnel.
- The "insecure server" approval is now remembered for the actual risky server, not the whole server list — if a re-imported config moves the no-cert-check flag to a different server, or two port-hopping servers share a host, you're asked again instead of the old approval silently transferring.
- Site diagnostic: DNS poisoning is detected even when the operator redirects blocked sites to a public block-page address (not just the classic localhost/private stub) — several different sites resolving to one identical address is the giveaway. The "Diagnose my server" probe is also more honest: when secure DNS is unavailable and a result can't be trusted, it says "DNS unclear" instead of declaring the server reachable.
- Fix: if part of an imported config fails to start, its secure-DNS route now switches to a surviving server instead of silently going outside the tunnel (where the ISP can see and block it).
- Fix: sharing "for any app" when your profiles are whole configs no longer copies an empty link with a green "copied" — it explains those need the "With my settings" form.
- Fix: a cloud-sync (WebDAV) restore with a damaged backup now reports an error instead of silently doing nothing; renaming a server to an empty or already-taken name now says so instead of silently closing.
- Fix (install/update): installing a new version over a running app now stops the app and all its cores first, so an in-place update can't fail on a locked file — most importantly winws.exe, which keeps a kernel driver loaded.
- Fix (uninstall): removing the app now also stops the server-less bypass engine (winws). Before, it kept running after uninstall — holding a network driver loaded, locking its own files against deletion, and still altering your live traffic. It's now shut down with the rest of the cores before files are removed.
- Fix: switching from system-proxy to full-device (TUN) mode while connected now clears the old system proxy instead of leaving Windows pointed at a port that's no longer listening (which made proxy-aware apps fail to connect until you toggled the proxy off by hand).
- Diagnostics are steadier: "is the network even up?" is now decided from several major Russian sites instead of a single one, so one slow/flaky site can't make a real block look like "you're offline"; and DNS-poisoning detection resets cleanly between runs so a normal CDN can't accumulate into a false alarm.
- "Share for any client" now works for an imported full config too: it extracts the individual servers inside it (your `🌍 VPN` becomes one standard link per server / a subscription), instead of failing because a whole config has no single-link form.
- A new-version notice now greets you on the Home screen the moment you open the app — not only if you dig into Settings → About — with a one-tap button to the signed release page. (The app still never auto-downloads or runs an installer for you: fetching and executing a file over a possibly-tampered network is the exact supply-chain attack this app defends against, so the download stays in your hands.) Dismiss it and it returns on the next launch until you've updated.
- Split-tunnel is easier to set up: pick an app straight from disk with a Browse button (no need to know its exact .exe name), or tap a common-app suggestion (Telegram, Chrome, Discord, Steam, …).
- The one-tap "enable server-less bypass / restart as admin" action in the diagnostic is now a clear, centered blue button instead of a muted one.
- Fix: pasting a "with my settings" share link (vpn://share…) into the manual import box now works — before, only the clipboard banner and drag-drop understood it, so a pasted link reported "nothing recognized". And those links are now much smaller: a whole-setup share is compressed, so a 5 KB link drops to roughly 1.5 KB and pastes (and scans as a QR) far more reliably. Older links still import unchanged.
- Fix: importing an AmneziaWG server carried inside a full config now classifies and bridges it correctly (it was treated as plain, blocked WireGuard); and a damaged or hostile "with my settings" share can no longer abort the import half-way — every shared setting is validated, and a bad one is ignored instead of stranding the import.
- Fix: the official release/installer now actually ships the server-less DPI-bypass engine (the WinDivert packet engine and its QUIC decoy) — previous packaging could leave it out, so the headline bypass showed as "not installed" no matter what; the build now refuses to package without it.
- Fix: the "insecure server" trust you grant is now tied to the actual server (its address), not its display name — so a subscription that rotates its exit behind the same name, or a re-imported server, asks again instead of silently inheriting the old approval.
- Fix (server-less bypass): a leftover packet-engine process from a previous run is now cleaned up on the next launch, so it can't keep desyncing your live web traffic with a stale site-list; and the bypass now restores its self-heal budget once it has run cleanly for a while, so it keeps recovering from antivirus/driver kills even if you never press Connect.
- Fix: the server-pool health badge now counts each member by its own warm latency, so a healthy pool no longer shows red "0 alive" (which also stops the app from needlessly re-pinging every server, ~45s of foreign handshakes, on each connect).
- Fix (Diagnose my server): more honest verdicts in exactly its target case — it no longer cries "whitelist collapse" when the Wi-Fi is simply off (it says Offline), a server that actually answers is reported reachable even if the public test IPs are blocked, and in TUN mode (which would capture the raw probe) it tells you to disconnect or use proxy mode instead of giving a wrong result. A slow-but-alive server on a congested mobile uplink now gets more time before being called blocked, and a server given by hostname is resolved over secure DNS so a poisoned or placeholder answer can't aim the probe at the wrong address.
- Under the 2025 "16KB connection-freeze" (which silently stalls a single long TLS connection to a foreign server), an automatic transport hop now prefers a freeze-immune transport (QUIC / XHTTP) over another equally-frozen Reality+Vision stream of the same survivability tier.
- Toggling "Hard-network mode" no longer needlessly reconnects when anti-DPI fragmentation is already on — nothing in the tunnel actually changes, so it stays up.
- A custom routing rule can no longer accidentally re-route the app's own network-health probe into the tunnel (e.g. a "public DNS → proxy" rule), which had let a genuine server block look like a whitelist collapse.
- Hardened cloud sync (WebDAV) against one more address trick: an IPv4-in-IPv6 literal (like ::127.0.0.1) can no longer slip past the internal-host guard and send your backup to a loopback/internal service.
- Cleaner, plainer settings: the TUN-only options (block-on-drop, faster DNS) are now greyed out and non-interactive when you're in system-proxy mode, instead of flipping to a green "on" that quietly does nothing; setting titles dropped the internal engine names (WinDivert / FakeIP / uTLS) and their descriptions are shorter and in plain language.
- Smoother first steps: the "add server" menu now leads with the everyday actions (paste a link, scan a QR, a subscription) and pushes the power tools below a divider; the Home tab icon is a shield, no longer the same power symbol as the big Connect button; and the empty-profiles hint tells you how to add your first server.
- One-tap import from the clipboard: copy a server link (vless://, vmess://, hysteria2://, a vpn:// share, …) and the Home screen offers a single "add this" banner — no more hunting for where to paste it. It only ever OFFERS (never auto-applies) and still shows the server preview before anything changes; it ignores ordinary web links and won't re-ask for the same one.
- "Works on Wi-Fi but not on mobile?" — when the tunnel goes dark on a selectively-blocking operator network, Home now offers a single "Make it work" button that turns on the full hard-network escalation (force TLS fragmentation + the survivor-preferring transport cascade + auto-adapt) in one tap, instead of three switches buried in settings. It appears only when it can actually help, and not on a true network shutdown (where it explains the physics limit instead).
- Gentler first run: the app no longer opens with a protection-mode question before you've added a single server — you land straight on Home, add a server, and the "full-device (TUN) vs app-proxy" choice is offered after your first successful connection instead.
- You'll learn about a new version even when GitHub itself is blocked: the update check now falls back to the (signed-in-spirit, data-only) censorship-facts feed for the latest version number. It only ever NOTIFIES and sends you to the official signed release page — it never downloads or runs anything itself, by design (auto-running a file fetched over a possibly-tampered network is exactly the risk this app exists to avoid).
- Activity → Policies: the "Test all" button now also connects you to the FASTEST reachable server in the group once it has measured them — the "test & connect fastest" the leading clients have. An insecure (no-cert-check) fastest server is named rather than silently switched onto, so the MITM consent is never bypassed.
- Activity now shows a small live download/upload graph under the speed numbers — an at-a-glance confirmation that traffic is actually flowing, drawn from the meter that was already running (no extra cost).

1.0.2 (05.06.26)

- Fix: switching between System proxy and TUN while connected now takes effect immediately. Before, the toggle only applied on the next manual reconnect, so the tunnel kept its old mode — apps that ignore the system proxy (Telegram, CLI tools like the OpenAI Codex CLI, Claude desktop) only work in TUN, so flipping to TUN appeared to "do nothing". The same now applies to routing mode, anti-DPI, fingerprint, kill-switch and split-tunnel changes.
- Fix: the "insecure server" warning is now shown ONCE per server, not on every connect — confirm it once and it's remembered.
- Fix: "Whitelist mode" no longer false-triggers in TUN. The foreign-reachability probe now goes direct (out your real uplink) instead of being swallowed by the tunnel, so the banner only appears on a genuine state-allowlist collapse — and that direct probe is now limited to the app's own check, so traffic from other apps to public DNS resolvers (8.8.8.8, 9.9.9.9, …) still rides the tunnel and your real IP never leaks out the physical connection.
- New: server-less DPI bypass (WinDivert) — an optional packet engine (zapret's winws) desyncs the outgoing TLS handshake (fake decoy + segment reorder + TTL/ACK fooling) so the censor can't read the site name (SNI), unblocking throttled / TLS-DPI sites (YouTube, Discord, Rutracker, LinkedIn, X…) with NO server at all (Settings → advanced). The default preset was tuned against a live Russian operator: fake decoy + multi-disorder at three SNI-targeted cut points + "datanoack" fooling, and the decoy now carries a padded Russian-government SNI (gosuslugi.ru) so even a stateful censor that tracks connection state classifies the flow as benign — verified to flip YouTube, X, Discord, LinkedIn and Rutracker from blocked to open in one shot. Covers QUIC / HTTP-3 too (a real fake-QUIC decoy), so YouTube/Google over HTTP-3 isn't left throttled. Needs administrator (loads a network driver) and the engine binary (fetched separately, like xray); pick among desync presets to find what survives your operator — and switch the method on the fly without dropping your connection. It does NOT help purely IP-blocked sites (Telegram) — those still need a server. Plain TLS fragmentation alone was dropped because ТСПУ reassembles it; this is the fake+disorder escalation that survives reassembly.
- Smarter failover: when a path is blocked, the transport cascade now prefers the transports that actually survive 2026-era blocking (XHTTP-split / Hysteria2-QUIC / Reality) and steers away from signature-blocked ones (plain VLESS — including VLESS tunnelled over WebSocket/gRPC, since only the Reality/XHTTP-wrapped forms survive in Russia — plus Shadowsocks / WireGuard), instead of hopping blind. The import preview warns about these dead transports too.
- Seamless switching: changing node, network (Wi-Fi↔Ethernet/wake) or a setting no longer flashes a red "Connecting…" — the button stays a calm amber "Checking…" because the proxy stays pinned and your traffic never actually drops during the swap.
- New: "Hard-network mode" for mobile operators (Settings → advanced) — Russian mobile networks throttle and block far harder than home Wi-Fi, so this one switch forces TLS fragmentation on and keeps the survivor-preferring transport cascade active regardless of your other settings, and the choice is remembered across restarts. Turn it on when a server works on Wi-Fi but not on mobile data.
- New: custom routing rules — force any domain or IP through the tunnel, keep it direct, or block it; your rules win over Smart routing AND over per-app split-tunnel (Settings → advanced). A malformed entry (a typo'd IP/CIDR — a bad prefix, a leading-zero or a leading-sign octet) is rejected as you add it instead of silently doing nothing or taking the connection down on the next start. Parity with the leading clients (Throne / Karing / v2rayN / Hiddify).
- New: cloud sync (WebDAV) — back up and restore your whole profile set to your own WebDAV cloud (Nextcloud, Koofr, box.com, …), so a reinstall or a second device is one tap away; a restore is size-capped, so a corrupt or oversized file on the server can't bloat the app's memory. Hardened for a censorship threat model: HTTPS-only, never follows a redirect/downgrade that would leak your password or the credential bundle, rejects internal/loopback hosts, and never surfaces the raw URL in an error; a credential pasted with a stray trailing space or newline still authenticates instead of failing with a puzzling 401. Parity with Karing's sync.
- New: optional faster DNS in TUN (FakeIP) — answers apps instantly with a placeholder address (for both IPv4 and IPv6 lookups) and resolves the real one at the exit, cutting first-load lag and avoiding DNS leaks (Settings → advanced; experimental, off by default).
- Imported XHTTP servers now keep their own split-tuning (the server's anti-freeze settings apply) instead of silently falling back to defaults.
- Import preview now warns when a server uses a transport that's widely blocked in Russia (plain WireGuard / Shadowsocks) and points you to Reality / Hysteria2 / XHTTP.
- WireGuard / AmneziaWG imports now keep the connection alive properly (PersistentKeepalive is honoured, and defaults to 25s when a config disables it).
- Hardened the XHTTP (xray) bridge: a server config xray rejects no longer becomes a silently-dead node — it's validated before use, the bad node is dropped so the rest of your pool keeps working (any Telegram or custom routing rule that pointed at the dropped node is moved to a surviving server instead of crashing the tunnel at startup), and if nothing usable is left you get a clear error instead of a dead or all-direct tunnel.
- Per-server ping now fills in for EVERY server, not just the fast-handshaking ones: slow-to-warm transports (XHTTP / Reality) are retried for a bit instead of being left showing "—" after one timed-out probe.
- New: "Diagnose my server" (Activity → Diagnostics) — a staged, raw probe of your selected server that names WHERE a connection breaks on this network: a state whitelist (nothing foreign reachable), the server's IP/port blocked, or its IP/port reachable but the protocol killed by DPI (the classic "works on Wi-Fi, not on mobile"). Copy the report to share. UDP/QUIC servers are flagged as not-passively-probeable.
- Tidier settings: related switches are now grouped into a few cards instead of a long stack of identical single-toggle rows, and TUN-only options (kill-switch, faster DNS, per-app split-tunnel) now tell you they only apply in TUN mode when you're on system-proxy — so you're not flipping something that quietly does nothing.
- Custom DNS now rejects a malformed entry (a pasted URL or a typo) instead of silently breaking name resolution, and notes that you press Enter to apply it while connected.
- Server-pool health badge no longer briefly flashes a red "0 alive" while it's still measuring a freshly opened pool — it shows a neutral "measuring" state until the checks land.
