## Cascade hardening (audit of the L3 orchestrator) — 6 failure modes closed
Acting on a deep external review of the transport-cascade, closed the scenarios that separate a demo from a battle version:
- **Network-alive gate BEFORE the cascade** — confirmed the `_directNetworkUp()` (raw direct dial) gate sits ahead of the cascade, not only the fp-restart, so a Wi-Fi/captive-portal drop never burns all families + a restart. (The proactive path runs only while traffic flows, so it's inherently gated.)
- **Signature-block ≠ IP-block** — the cascade now counts probed-vs-answered untried families; if some existed but NONE answered, every transport on the server is dark at once = an IP/server block (not per-signature), so it STOPS instead of wasting fp-restarts and logs "rotate the server/relay" (the L0 hook). Server rotation itself needs the relay-pool infra.
- **Single owner during an episode** — `_probeHop` (auto-hop) now stands aside while a cascade/degradation episode is active (`_adapting || _triedTransports.isNotEmpty || _proactiveCooldown>0`), so the three control loops (urltest / auto-hop / cascade) can't race into a double-switch.
- **Episode lifecycle** — the tried-family set clears only on SUSTAINED recovery (3 healthy ticks ≈ 54 s), not the first good tick, so a family blocked early in a wave is re-tried later when the wave moves on.
- **Physical-layer diversity ordering** — candidates are ordered so a DIFFERENT L4 comes first (TCP↔QUIC): a TCP wave is answered by Hy2/QUIC, never another TCP transport. Verified live: xhttp(TCP-bridged) dark → hops to a QUIC (Hy2) node first.
- **Testability** — extracted the cascade DECISION into a pure `planCascade(groups, tried)` (resolution + diversity ordering + dead-end), unit-tested with 7 cases (diversity, IP-block dead-end, tried-exclusion, leaf resolution through nested groups, no-selector, GLOBAL-exclusion, single-member). The watchdog state-machine tests (network-down/episode-reset) still want a ClashApi mock — noted as follow-up.

## L3 orchestrator: proactive hop on DEGRADATION (the "Hydra" head-mover)
- The insight (vs Mullvad): ТСПУ kills one signature per wave, so any SINGLE transport is one wave from death — the edge is a client that MOVES to a different physics BEFORE it's fully caught. Reactive escalation only fired after ~54 s of total darkness. Now the watchdog, while the tunnel still carries traffic, tracks the active leaf's RTT against an EMA baseline; on SUSTAINED SEVERE degradation (>3× baseline AND >800 ms for 3 cycles ≈ 54 s) it triggers the transport-cascade hop EARLY — moving off a throttled/degrading path before it dies. Deliberately conservative + a ~3.5 min cooldown after a proactive hop so a noisy path can't churn (the bug we just fixed). HONEST LIMIT: the degradation THRESHOLDS can't be battle-verified headless (need a real degrading path) — the cascade they trigger is verified; the thresholds need field tuning. Watch for `proactive: path degrading … → hopped transport early` in the log.

Removed `detour: direct` from all four sites (a DNS server dials direct by default). **Verified against the user's real inputs:** their `vless://…@xx.xx.xx.xx` Reality link now passes `sing-box check` in Smart/Global/TUN/anti-DPI AND carries live traffic (direct `178.66.156.156` → tunnel `xx.xx.xx.xx`); the AmneziaWG `.conf` now imports + validates. Regression-locked (`test/connect_regression_test.dart` asserts no path emits `detour: direct`; `tool/probe_link.dart` re-checks the real link end-to-end).
- **Seamless long-term connection (was: reconnect churn).** `onNetworkChanged` did a full `restart()` on *every* Windows address-change event, fighting sing-box's own `auto_detect_interface` (which rebinds to the new route without dropping the tunnel) — visible as periodic reconnects + packet loss on games. ## Independent production audit — hardening (in progress)
Five independent adversarial auditors (config/anti-DPI, security/kill-switch, Dart robustness, UI/UX, release-eng) reviewed the whole app against a "best-in-RF, production-ready" bar. Their verified findings, fixed in batches. (False positives recorded too: the Smart/Global direct resolver is Yandex `77.88.8.8`, not Cloudflare — `1.1.1.1` is only ever used *through the tunnel*; latency *does* poll every 5s.)

### Batch 1 — security & data-integrity
- **Kill-switch no longer fails OPEN on a network change (BLOCKER).** A Wi-Fi/Ethernet/wake switch ran `restart()` → `stop()` which *restored the direct proxy* for the whole ~6s reconnect, leaking plaintext SNI/DNS onto the just-joined (hostile RF) network. `restart()` now keeps the system proxy pointed at the local port across the swap, so traffic fails **CLOSED** (blocked, never direct) until the new core is up; only a user-initiated Stop restores the real proxy. Terminal errors (bad config/exe) still restore connectivity so the user isn't stranded.
- **Clash API is now secret-guarded (HIGH).** It bound `127.0.0.1:9090` with no auth — any local process, or a web page via DNS-rebinding (no CORS on the REST endpoints), could `GET /connections` (every host you reach → deanonymization) and `PUT /proxies` (silently force your exit to an attacker's node). Every config now carries a cryptographically-random per-launch `secret`; `ClashApi` sends `Authorization: Bearer` on every request + the traffic WebSocket. **Proven against the live core:** `dart run tool/verify_store.dart --connect` now asserts `anon → 401, auth → 200`.
- **Orphan-core kill scoped to OUR PIDs (HIGH).** `taskkill /F /IM sing-box.exe` blanket-killed *every* sing-box/xray on the machine — including another running client (Hiddify/Throne bundle sing-box) or a second instance. Now we record each spawned PID and reap only those, gated on `IMAGENAME` (guards PID reuse).
- **Atomic, crash-safe profile/settings writes (BLOCKER).** `writeAsStringSync` truncates-then-writes; a crash/power-loss mid-write left a 0-byte store → every profile lost on next launch. Both stores now write temp+`flush`+`rename` (atomic on NTFS) via `CorePaths.atomicWrite`, and `ProfileStore.save` is guarded so a write failure can't crash a UI handler.
- **One bad node can't brick the whole store (HIGH).** A single hand-edited/corrupt entry threw inside the one-shot parse and discarded *every* profile (re-creating the "no servers" incident). Each node now parses in its own try — bad ones are skipped, the rest survive.
- **Settings test-isolation (BLOCKER — half-fixed before).** `ProfileStore` had `overrideDir` but `SettingsController` didn't, so a widget test that flipped a setting rewrote the REAL settings.json (and could arm TUN mode on the user's machine). Added `SettingsController.overrideDir`; unknown/newer enum values now default just that one field instead of resetting all settings.
- **Binary-planting fix (MEDIUM).** Core resolution checked `CWD/core/windows/<exe>` *before* the exe-relative path, so launching from an attacker-controlled CWD could run a planted sing-box/xray. Exe-relative is now resolved FIRST (CWD only as a dev fallback). `runtimeDir()` no longer throws out of a UI handler on a locked-down box (falls back through TEMP → a unique temp dir).
- **Native proxy hardening.** Proxy/URL strings now convert UTF-8→UTF-16 via `MultiByteToWideChar` (the old `wstring(begin,end)` corrupts any non-ASCII value → a garbage registry ProxyServer = no internet); the backup guard now refuses to snapshot ANY loopback `ProxyServer` (a crash-stale `127.0.0.1:<old-port>`), not just an exact match, so a later restore can't strand the user on a dead port.
- **1.14 readiness signal.** Pre-flight now also runs `sing-box check` WITHOUT the `ENABLE_DEPRECATED_*` env; if the config only validates with the compat flags it logs a warning (it will hard-FATAL on sing-box 1.14) instead of shipping a silent time-bomb.
- 6 new regression tests (`test/hardening_test.dart`) lock in store resilience, atomic writes, settings isolation, and the Clash secret.

### Batch 2 — config/parser correctness
- **Reject un-handshakeable Reality nodes (HIGH).** A `vless://…security=reality` link (or Clash `reality-opts`) with no `pbk`/`public-key` was imported as a node that can never complete a handshake — surfacing to the user as an unexplained "no connection." Such links are now dropped at parse time.
- **IPv6-literal endpoints parse correctly (A2).** `_hostPort` split on the last `:`, mangling `[2001:db8::1]:51820` (WireGuard endpoints, bare-IPv6 ss nodes). It now unwraps brackets → bare IPv6 host + correct port, and treats an unbracketed multi-colon address as a port-less literal.
- **`ss://` SIP002 `plugin=` preserved (HIGH).** The parser stripped the query, silently turning an obfs-local / v2ray-plugin / shadow-tls node into a bare, easily-blocked Shadowsocks one. It now carries `plugin` + `plugin_opts` through to sing-box.
- **Subscriptions validate the HTTP response (C6).** `_fetch` returned the body regardless of status; a blocked/expired sub in RF (HTML error / captive portal / 4xx-5xx) got parsed as a node list (silent no-op, or a junk node from a stray `vless://` in an error page). Non-200 and HTML bodies now throw, surfacing the real failure.
- **Refresh can't crash after the sheet closes (C5).** A subscription fetch that completes after the profiles notifier is disposed no longer writes `state` (which throws in Riverpod 3.x) — it bails cleanly.
- **Canonical dedup key (C9).** `_contentKey` hashed `jsonEncode` of unordered maps, so a sub returning the same node with keys in a different order defeated dedup and re-added it as "Name (2)", "(3)", … every refresh. Keys are now sorted recursively before hashing.
- **Deleting the active node disconnects it (C10).** Removing the currently-connected profile left the core tunnelling through it (UI showed a different node than traffic used). `remove()` now restarts the live core onto the new selection, or stops it if none remain.
- **Smart mode degrades safely if rule-sets are missing (A8).** If the bundled `.srs` aren't on disk (packaging miss / AV quarantine), Smart mode now builds a rule-set-free config (all via proxy, private direct) instead of FATAL-ing on a missing local path — and a shipping build NEVER emits a remote (RF-blocked) rule-set.

### Batch 5 — second independent audit (re-audit of the hardening) + fixes
A fresh adversarial audit of the **changed** code (3 reviewers: kill-switch, config/parser, release-eng) — hunting for bugs the hardening *introduced*. It found real ones; all fixed:
- **Kill-switch: user Stop during a restart could leave you connected + briefly leak (HIGH).** A `stop()` landing while a network-change `restart()` was mid-swap could let the restart-spawned `start()` resume and silently reconnect (and flash DIRECT first). `start()` now honors a Stop that lands mid-start (aborts + restores), and `restart()` is wrapped in try/finally so its `_restarting` flag can never get stuck true.
- **Kill-switch: terminal start() errors re-opened the fail-open hole (HIGH).** The Batch-1 fix made every start() error restore the proxy — but during a network-change restart (or an auto-reconnect) that drops traffic to DIRECT on a hostile network. Restore is now gated on a `_proxyActive` flag: if a live tunnel was protecting traffic, a failed swap fails **CLOSED** (user presses Stop to restore); only a first/standalone connect restores.
- **`ss://` port lost to SIP002's `/` (HIGH).** My own Batch-2 plugin fix exposed it: `ss://…@host:port/?plugin=…` parsed the port as `port/` → defaulted to :443 → dead node. Now strips the trailing `/`. Regression-tested.
- **Installer was un-buildable + unsigned (HIGH).** `installer.iss` had a non-hex `AppId` GUID (`iscc` rejects it) → fixed to a valid GUID; `package.ps1` now builds the installer via `iscc` and **signs the setup.exe** too (it was only signing the zip's binaries), and a `-RequireSigning` flag fails the release (not just warns) if the cert is missing.
- **Rule-sets are now SHA-256-pinned too.** They control direct-vs-proxy routing, so a poisoned `.srs` is a leak — `fetch-cores.ps1` now verifies all three against pinned hashes (bumping one is a deliberate, re-tested change).
- **Smaller:** atomic writes fall back to an in-place write if the target is locked by AV/indexer (no silent save-loss); the PID file is cleared on a clean stop (no stale-PID accumulation / recycle mis-kill); tapping a node in the list now switches the live tunnel (so the running node always matches the selection); a network change while reconnecting now moves onto the new path; empty `ss` `plugin=` is ignored; wintun's `amd64` path match anchored.
- Auditors independently **VERIFIED OK**: the fail-closed invariant on the normal network-change path, `_onExit` ordering vs stop/restart, the Smart-mode degrade emits valid sing-box (no dangling rule-set refs), Reality-reject/IPv6/dedup correctness, the SHA-256 pins match the real bundled bytes, the pinned GitHub asset names exist, PowerShell-5.1 compatibility, and the native `Utf16FromUtf8`/`IsOwnLoopback`.

### Batch 5b — kill-switch final confirmation
A third, confirmation-only audit traced one last race the Batch-5 fix had left: a user **Stop landing in the `await SystemProxy.set` gap** could leave a "zombie" `running` state (no actual leak — the OS proxy ended cleared — but a later network change could silently reconnect). Fixed by setting `_proxyActive` before the await and **re-checking intent (`!_autoReconnect` / `_proc==null`) after it**, before committing to `running`; the final commit is now straight-line with no await, closing the TOCTOU.
